## Smart Contract Layer ‚Äî Base Sepolia

The Base Sepolia layer handles payment execution and compliance metadata collection. It consists of these main contracts.

### SmartWallet

The SmartWallet is an ERC-4337 compliant smart account. Each business entity deploys exactly one smart wallet, created as a minimal proxy clone via the SmartWalletFactory using `Clones.cloneDeterministic`. The deterministic address means a wallet's address is known before deployment, enabling counterfactual wallets and gasless `initCode` deployment.

Key responsibilities:
- Validates user operations via `validateUserOp` using EIP-191 message hashing
- Executes single and batch native transfers
- Manages committed vs. available balance to prevent double-spending on scheduled payments
- Exposes `executeBatchIntentTransfer` ‚Äî callable only by the IntentRegistry ‚Äî for scheduled payment execution

**Fund commitment model.** When a recurring payment intent is created, `increaseCommitment(totalCommitment)` is called on the wallet. The committed amount is subtracted from available balance, preventing the funds from being spent elsewhere. Each executed cycle calls `decreaseCommitment` by the cycle amount. Cancellation releases all remaining committed funds.

### SmartWalletFactory

The factory deploys and tracks smart wallet clones. It accepts a single `owner` address, derives a deterministic salt, and deploys the clone using OpenZeppelin's `Clones` library.

For testnet purposes, the factory is configured with a `MockUSDC` drip so newly onboarded wallets can receive stablecoin funds for simulated treasury flows.

The factory also triggers automatic company registration on the ComplianceRegistry at wallet creation time. This establishes the identity link mapping the wallet proxy address to the owner's EOA at the earliest possible moment.

### IntentRegistry

The IntentRegistry is the scheduling and compliance dispatch engine. It manages the full lifecycle of recurring payment intents.

**Creating an intent.** The caller provides:
- `name` ‚Äî a human-readable label
- `recipients[]` and `amounts[]` ‚Äî matched arrays; max 10 recipients per intent
- `duration` ‚Äî total active time in seconds
- `interval` ‚Äî time between executions in seconds (minimum 30s)
- `transactionStartTime` ‚Äî Unix timestamp for first execution (0 = immediate)

The registry validates fund availability, calculates total commitment (`amountPerCycle √ó totalCycles`), locks funds via `increaseCommitment`, and dispatches a compliance report at creation time. For recurring payments, each execution should ideally produce a corresponding compliance record referencing the originating intent.

**Executing an intent.** The registry implements `checkUpkeep` and `performUpkeep`, the Chainlink Automation interface. Automation nodes call `checkUpkeep` and trigger `performUpkeep` when execution conditions are met. Execution is skipped (not reverted) if a recipient transfer fails, with the failed amount tracked in `intent.failedAmount` for recovery.

**Cancelling an intent.** The wallet owner can cancel any active intent at any time. The remaining committed funds are released immediately.

### ComplianceRegistry

The ComplianceRegistry is the core privacy-preserving storage contract. It maintains an isolated ledger per company, keyed by the wallet proxy address.

Each compliance record stores:
- `txHash` ‚Äî the deterministic link to the transaction or intent execution
- `recipients[]` and `amounts[]` ‚Äî plaintext payment data. Payment amounts and recipient addresses remain public on the compliance ledger because they already exist on the public payment ledger. Only the compliance context is encrypted.
- `encryptedPayload` ‚Äî AES-256-GCM ciphertext encapsulating categories and jurisdictions
- `timestamp` ‚Äî block timestamp at record creation

Auditors and company owners maintain the encryption keys required to read from this contract. 

### Sponsorship Model

Complyr uses Pimlico's ERC-4337 bundler and paymaster services for gas sponsorship on Base Sepolia. Sponsorship configuration is handled in the web client layer (`bundler.ts` and `smartAccountClient.ts`) rather than through a custom onchain paymaster contract in this repository.
