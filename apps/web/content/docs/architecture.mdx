## Overview

Complyr is a monorepo with four active parts:

1. A Next.js web application in `apps/web`
2. Solidity contracts in `packages/contracts`
3. An Envio indexer in `packages/indexer`
4. A PostgreSQL-backed contacts store used by the web app

The codebase follows a standard pattern:

- the web app handles UX, orchestration, and client-side encryption;
- the contracts are the source of truth for payments and audit records;
- the indexer builds a GraphQL activity feed from contract events;
- PostgreSQL stores app-owned data that does not belong onchain.

## Component Responsibilities

### Web App

The web app is responsible for:

- authenticating users with Privy;
- creating or connecting a smart account;
- submitting single, batch, and recurring payment flows;
- encrypting audit inputs in the browser with the Zama relayer SDK;
- querying indexed wallet activity from Envio;
- reading encrypted audit records directly from `AuditRegistry`;
- managing contacts through server routes backed by PostgreSQL;
- serving the documentation site.

Important modules:

- `src/app/provider.tsx`
- `src/lib/SmartAccountProvider.tsx`
- `src/lib/customSmartAccount.ts`
- `src/hooks/payments/*`
- `src/lib/fhe-audit.ts`
- `src/hooks/useAuditLogs.ts`
- `src/lib/envio/client.ts`

### Smart Contracts

The contracts are the canonical backend of the product.

**`SmartWallet`**

- ERC-4337 account implementation
- executes direct and batch payments
- executes registry-triggered recurring transfers
- records audit entries on successful payment flows

**`SmartWalletFactory`**

- deploys deterministic wallet clones
- registers each wallet in `AuditRegistry`
- can seed new wallets with ETH and a configured token drip

**`IntentRegistry`**

- stores recurring payment intents
- tracks committed wallet funds
- exposes Chainlink Automation-compatible execution methods
- triggers scheduled transfers through the wallet

**`AuditRegistry`**

- stores encrypted audit records
- stores encrypted rollups for reporting
- manages company master wallets and reviewer access
- stores reviewer threshold tests and result queues

**`MockUSDC`**

- test token used by demo flows on Sepolia

### Indexer

The Envio indexer listens to selected contract events and materializes a GraphQL read model.

It currently maintains:

- `Wallet`
- `Transaction`
- `Intent`

The frontend uses this for dashboard activity and transaction history. It does not replace direct chain reads for encrypted audit data.

### PostgreSQL

The web app uses Drizzle ORM with PostgreSQL for offchain application data.

The active product-facing schema here is the contacts subsystem:

- contacts
- contact addresses

These records are accessed through the Next.js API routes under `/api/contacts`.

## Data Classification

### Public Onchain Data

The following data is visible onchain:

- wallet addresses;
- recipients;
- token addresses;
- timestamps and transaction hashes;
- plaintext reference IDs stored in audit records.

### Encrypted Onchain Data

The following values are encrypted before they are written onchain:

- audit copy of payment amounts;
- category codes;
- jurisdiction codes;
- reviewer thresholds;
- reviewer result signals;
- encrypted rollups used for reporting and checks.

## Main Flows

### Wallet Deployment

1. The user logs in with Privy.
2. The app predicts or deploys a wallet through `SmartWalletFactory`.
3. The factory registers the new wallet with `AuditRegistry`.
4. The frontend initializes an ERC-4337 client for future operations.

### Single and Batch Payments

1. The user enters recipients, amounts, and audit metadata.
2. The browser encrypts the audit payload with the Zama SDK.
3. The frontend submits a UserOperation through Pimlico.
4. `SmartWallet` executes the payment.
5. `SmartWallet` writes the encrypted audit record to `AuditRegistry`.
6. The indexer captures emitted events for the activity feed.

### Recurring Payments

1. The user creates a recurring payment intent.
2. The app encrypts the audit data in the browser.
3. `IntentRegistry` stores the schedule and reserves committed funds.
4. A keeper-compatible caller executes the next due payment.
5. The wallet performs the transfer and audit data is recorded.
6. The indexer updates intent and transaction history.

### Audit and Reviewer Access

The audit surfaces read directly from `AuditRegistry`.

- the business audit dashboard loads ledger records and decrypts them client-side when the connected wallet is authorized;
- the auditor portal verifies reviewer access onchain, creates encrypted threshold tests, reads reviewer results, and decrypts permitted data client-side.

## Integrations

Complyr depends on these external systems:

- Privy for authentication and embedded wallets
- Pimlico for bundling and paymaster support
- Ethereum Sepolia RPC endpoints
- Zama FHE relayer tooling
- PostgreSQL for app-managed data

## Why The Split Looks This Way

This architecture keeps the responsibilities clean:

- settlement, commitments, and audit records live onchain;
- dashboard history is optimized through an indexer;
- contacts and app-only records stay in a relational database;
- encryption and decryption authorization remain in the client where the user controls the signing wallet.
